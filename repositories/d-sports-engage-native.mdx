## Overview

**d-sports-engage-native** (package name: `engage-native`) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android: wallet, shop, leaderboard, locker room, and profile. The app also runs as a responsive PWA on desktop web.

## Tech stack

| Category       | Technology                                  |
| -------------- | ------------------------------------------- |
| Framework      | Expo 54, React Native 0.81, React 19        |
| Auth           | Clerk (`@clerk/clerk-expo`)                  |
| Payments       | RevenueCat (`react-native-purchases`)        |
| Web3           | Thirdweb SDK                                 |
| State          | Zustand + MMKV persistence                   |
| UI             | Lucide React Native                          |
| Navigation     | Expo Router 6 (file-based routing)           |
| Monitoring     | Sentry (`@sentry/react-native`)              |
| Push           | OneSignal (`react-native-onesignal`)         |
| Package manager| Bun 1.3.9                                    |

## Features

- **Wallet** — tokens, holdings, pack opening, crypto checkout (Arbitrum, Ethereum, Polygon)
- **Shop** — collectibles, cart, coin bundles, RevenueCat fiat checkout, and crypto checkout
- **Leaderboard** — global and team-scoped rankings with filters
- **Locker room** — social feed, daily pick-em, spin wheel, guess-the-player games, quests
- **Profile** — user profile, achievements, followers/following, points history
- **Settings** — account, appearance, notifications, privacy, accessibility, in-app changelog
- **Theme** — dark/light mode (default dark)

- Reward presentation differentiates free/unpaid states and claimability.

- Team discovery, favorite-team interactions, and leaderboard context are integrated across locker room and profile surfaces.

## Getting started

<Steps>
  <Step title="Install dependencies">
    ```bash
    bun install
    ```
  </Step>
  <Step title="Configure environment variables">
    Create a `.env` file at the project root. Only `EXPO_PUBLIC_*` keys are accessible at runtime.

    | Variable | Purpose |
    | -------- | ------- |
    | `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` | Clerk authentication |
    | `EXPO_PUBLIC_API_URL` | Backend API base URL (defaults to `https://api.d-sports.org`) |
    | `EXPO_PUBLIC_TW_CLIENT_ID` | Thirdweb client ID for Web3 features |
    | `EXPO_PUBLIC_REVENUECAT_API_KEY` | RevenueCat API key for in-app purchases |
    | `EXPO_PUBLIC_REVENUECAT_APPSTORE_ID` | RevenueCat App Store ID |
    | `EXPO_PUBLIC_REVENUECAT_ENTITLEMENT` | RevenueCat entitlement identifier |
    | `EXPO_PUBLIC_SUPABASE_URL` | Supabase project URL |
    | `EXPO_PUBLIC_SUPABASE_KEY` | Supabase publishable key |
  </Step>
  <Step title="Start the development server">
    ```bash
    bunx expo start
    ```
    Press **a** for Android, **i** for iOS, or scan the QR code with Expo Go.
  </Step>
</Steps>

## Architecture

### Screen pattern

Screen files under `app/(tabs)/` contain **only JSX**. All state, effects, and handlers live in dedicated hooks:

- `hooks/use-wallet-screen.ts` — wallet and token state, PIN verification, transactions
- `hooks/use-shop-screen.ts` — cart state, product queries, checkout logic

Sub-components are extracted into `components/wallet/` and `components/shop/` with barrel exports.

### API client layer

The API client in `lib/api/` provides authenticated requests using Clerk tokens. All requests follow a normalized `{ success, data?, error?, code? }` response envelope.

```typescript
import { useApi } from "@/lib/api";

function MyComponent() {
  const api = useApi();
  const result = await api.user.getProfile();
}
```

The `useApi()` hook exposes domain modules: `user`, `quests`, `leaderboard`, `wallet`, `lockerRoom`, `teams`, `collectibles`, `games`, `shop`, and `checkout`.

### State management

- **Zustand** store (`services/store.ts`) with MMKV persistence for theme, cart, and points
- **React Context** for auth/user (`context/user-context.tsx`), collectibles, and navbar visibility
- **MMKV cache** (`lib/api/cache.ts`) provides cache-first fetching with fallback

### Checkout and payments

- **RevenueCat** handles fiat payments: Apple IAP (native), Google Play (native), Stripe (web)
- **Crypto checkout** calls the PWA backend via `POST /api/checkout/crypto` and `POST /api/checkout/crypto/verify`
- **Thirdweb SDK** signs on-chain transactions on Arbitrum (default), Ethereum, or Polygon

### Path alias

All imports use `@/*` which maps to the project root:

```typescript
import { useUser } from "@/context/user-context";
import type { Token } from "@/types/wallet.types";
```

## EAS build profiles

Builds use [EAS Build](https://docs.expo.dev/build/introduction/) with these profiles defined in `eas.json`:

| Profile | Channel | Distribution | Notes |
| ------- | ------- | ------------ | ----- |
| `development` | `development` | Internal | Dev client, iOS simulator enabled |
| `development-device` | `development` | Internal | Dev client on physical devices |
| `preview` | `preview` | Internal | Android APK for testing |
| `staging` | `staging` | — | Auto-increment version |
| `production` | `production` | — | Auto-increment version |

### Common build commands

```bash
# Development build (all platforms)
bun run build:dev

# Preview build (Android APK)
bun run build:preview:android

# Production build (iOS)
bun run build:prod:ios

# OTA update
bun run update

# Submit to stores
bun run submit:ios
bun run submit:android
```

### OTA updates

A GitHub Actions workflow (`.github/workflows/eas-update.yml`) publishes OTA updates via `eas update` on pushes to `develop` and `main` branches.
