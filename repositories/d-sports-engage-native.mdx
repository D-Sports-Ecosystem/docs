## Overview

**d-sports-engage-native** (package name: `engage-native`, v1.10.10) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android: wallet, shop, leaderboard, locker room, and profile.

The app targets both native and web (responsive) and uses the same backend (`d-sports-api`) as the PWA for API and checkout flows.

## Features

- **Wallet** — tokens, holdings, pack opening, crypto checkout (via PWA backend)
- **Shop** — collectibles, cart, coin bundles, checkout (fiat via RevenueCat, crypto via Thirdweb)
- **Leaderboard** — rankings and filters with themed team views
- **Locker room** — social feed, quests, daily games, D-Sports Cash
- **Profile** — user profile, settings, achievement badges
- **Theme** — dark/light mode (default dark)

## Getting started

<Steps>
  <Step title="Install dependencies">
    ```bash
    bun install
    ```
  </Step>
  <Step title="Configure environment variables">
    Create a `.env` file in the project root with the following keys:

    | Variable | Description |
    | -------- | ----------- |
    | `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` | Clerk publishable key for authentication |
    | `EXPO_PUBLIC_API_URL` | Backend API base URL (e.g. `https://api.d-sports.org`) |
    | `EXPO_PUBLIC_TW_CLIENT_ID` | Thirdweb client ID for web3 features |
    | `EXPO_PUBLIC_REVENUECAT_API_KEY` | RevenueCat API key for in-app purchases |
    | `EXPO_PUBLIC_REVENUECAT_APPSTORE_ID` | RevenueCat App Store ID |
    | `EXPO_PUBLIC_REVENUECAT_ENTITLEMENT` | RevenueCat entitlement identifier |
    | `EXPO_PUBLIC_SUPABASE_URL` | Supabase project URL (client-side features) |
    | `EXPO_PUBLIC_SUPABASE_KEY` | Supabase publishable anon key |

    <Note>Only `EXPO_PUBLIC_*` keys are accessible at runtime in the Expo app.</Note>
  </Step>
  <Step title="Start the development server">
    ```bash
    bunx expo start
    ```
    Press `a` for Android, `i` for iOS, or scan the QR code with Expo Go.
  </Step>
  <Step title="Run a type check">
    ```bash
    bun tsc --noEmit
    ```
  </Step>
</Steps>

## Architecture

### Modular screen pattern

Screen files in `app/(tabs)/` contain **only imports and JSX**. All state, effects, and handlers live in dedicated hooks:

- `hooks/use-wallet-screen.ts` — wallet/token fetch, PIN verification, transaction handlers
- `hooks/use-shop-screen.ts` — cart state, product queries, carousel auto-scroll, checkout logic

Sub-components are extracted into `components/<domain>/` directories with barrel exports via `index.ts`.

### State management

- **Zustand** store in `services/store.ts` with MMKV persistence for theme, cart, and points
- **React Context** providers in `context/` for authentication (`UserContext`), collectibles (`CollectiblesContext`), and navbar visibility

### API client layer

The API client in `lib/api/` provides domain-specific modules (wallet, shop, user, quests, leaderboard, locker room, teams, collectibles, checkout) accessed through the `useApi()` hook in `lib/api/index.ts`. All requests use auth token injection via `lib/api/client.ts` with MMKV cache-first fetching in `lib/api/cache.ts`.

### Checkout and payments

| Channel | Provider | Details |
| ------- | -------- | ------- |
| Fiat (native) | RevenueCat | Apple IAP, Google Play |
| Fiat (web) | RevenueCat + Stripe | Web checkout |
| Crypto | Thirdweb SDK | On-chain via PWA backend (`POST /api/checkout/crypto`) |

Supported chains: Arbitrum (default), Ethereum, Polygon.

### Path aliases

All imports use the `@/*` alias which maps to the project root:

```typescript
import { useUser } from "@/context/user-context";
import type { Token } from "@/types/wallet.types";
```

## EAS builds

The project uses [EAS Build](https://docs.expo.dev/build/introduction/) with four profiles defined in `eas.json`:

| Profile | Use case |
| ------- | -------- |
| `development` | Simulator builds with dev client |
| `development-device` | Physical device builds |
| `preview` | Internal testing builds |
| `production` | App Store / Play Store submissions |

Common build commands:

```bash
bun run build:dev        # Development build
bun run build:preview    # Preview build
bun run build:prod       # Production build
```

OTA updates are handled via `eas update` on the `develop` and `main` branches through CI.
