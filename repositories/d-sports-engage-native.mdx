## Overview

**d-sports-engage-native** (package name: `engage-native`, version `1.10.10`) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android: wallet, shop, leaderboard, locker room, and profile.

- **Run:** `bunx expo start` or `bun run start` — then press `a` for Android or `i` for iOS, or scan the QR code with Expo Go.

## Features

- **Wallet** — Tokens, holdings, pack opening, crypto checkout (via PWA backend)
- **Shop** — Collectibles, cart, coin bundles, checkout
- **Leaderboard** — Rankings and filters
- **Locker room** — Social feed, quests, daily pick-em, spin wheel, and guess-the-player games
- **Profile** — User profile and settings
- **Theme** — Dark/light mode (default dark)
- **Onboarding** — New-user onboarding flow

```text
├── wallet/              # Wallet sub-components (TokenRow, ActionModal, PackOpeningModal, etc.)
├── shop/                # Shop sub-components (CartModal, CryptoCheckoutModal, etc.)
├── locker-room/         # Locker room components (feed, quests, games, teams)
├── leaderboard/         # Leaderboard and modal components
├── settings/            # Settings items, sections, and modals
├── ui/                  # Reusable UI primitives (Button, TextField, WebHoverWrapper)
├── Icon/                # Icon wrapper using lucide-react-native
└── theme-provider.tsx   # Theme context (dark/light)

hooks/                   # Screen-level hooks (use-wallet-screen, use-shop-screen, etc.)
lib/api/                 # API client modules (wallet, shop, user, quests, checkout, etc.)
context/                 # React Context providers (user, collectibles, navbar visibility)
services/                # Zustand store, MMKV storage, types
theme/                   # Brand colors, spacing, typography tokens
types/                   # Shared TypeScript types (wallet, shop, checkout, API)
constants/               # Shared constants (wallet, shop, locker room)
```

## Architecture

- **Modular screens** — Screen files contain only JSX. All state, effects, and handlers live in dedicated hooks (`use-wallet-screen.ts`, `use-shop-screen.ts`).
- **Extracted sub-components** — Wallet and shop screens are decomposed into `components/wallet/` and `components/shop/` with barrel exports.
- **API client layer** — `lib/api/client.ts` provides a base HTTP client with auth token injection. Domain modules handle wallet, shop, user, quests, leaderboard, locker room, teams, collectibles, and checkout calls.
- **Cache fallback** — `lib/api/cache.ts` uses MMKV for cache-first fetching.
- **Crypto checkout** — Thirdweb SDK signs on-chain transactions via the PWA backend. Supported chains: Arbitrum (default), Ethereum, and Polygon.
- **Zustand + MMKV** — Global state with synchronous persistence.
- **File-based routing** — Expo Router with route groups `(tabs)`, `(auth)`, `(onboarding)`, and typed routes.
- **Path alias** — `@/*` maps to the project root.
- **PWA-ready** — `display: standalone`, responsive desktop layout (`maxWidth: 480px`), web hover states, and keyboard navigation.

## Environment variables

The app uses `EXPO_PUBLIC_*` keys, which are accessible at runtime in the Expo app:

| Variable | Purpose |
| -------- | ------- |
| `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` | Clerk authentication |
| `EXPO_PUBLIC_API_URL` | API backend base URL |
| `EXPO_PUBLIC_TW_CLIENT_ID` | Thirdweb client ID (Web3) |
| `EXPO_PUBLIC_REVENUECAT_API_KEY` | RevenueCat API key |
| `EXPO_PUBLIC_REVENUECAT_APPSTORE_ID` | RevenueCat App Store ID |
| `EXPO_PUBLIC_REVENUECAT_ENTITLEMENT` | RevenueCat entitlement name |
| `EXPO_PUBLIC_SUPABASE_URL` | Supabase project URL |
| `EXPO_PUBLIC_SUPABASE_KEY` | Supabase publishable key |

- Quest UI now reflects team-scoped eligibility and pass-gated progression where applicable.
- Reward presentation differentiates free/unpaid states and claimability.

- Team discovery, favorite-team interactions, and leaderboard context are integrated across locker room and profile surfaces.

## EAS builds

The project uses [EAS Build](https://docs.expo.dev/build/introduction/) for native builds. Profiles are defined in `eas.json`:

| Profile | Description | Command |
| ------- | ----------- | ------- |
| `development` | Dev client with simulator support | `bun run build:dev` |
| `preview` | Internal distribution (APK on Android) | `bun run build:preview` |
| `staging` | Auto-incrementing staging channel | via EAS CLI |
| `production` | Auto-incrementing production channel | `bun run build:prod` |

You can also build per platform — for example, `bun run build:dev:ios` or `bun run build:prod:android`.

The app targets both native and web (responsive) and uses the same backend (d-sports-api) as the PWA for API and checkout flows.
