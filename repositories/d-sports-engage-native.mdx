## Overview

**d-sports-engage-native** (package name: `engage-native`, version **1.14.2**) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android: wallet, shop, leaderboard, locker room, and profile.

- **Run:** `bunx expo start` or `bun run start` — then press `a` for Android or `i` for iOS, or scan the QR code with Expo Go.

## Tech stack

| Category       | Technology                                   |
| -------------- | -------------------------------------------- |
| Framework      | Expo 54, React Native 0.81, React 19         |
| Auth           | Clerk (Expo)                                 |
| Payments       | RevenueCat (react-native-purchases)          |
| Web3           | Thirdweb                                     |
| State          | Zustand                                      |
| Storage        | MMKV                                         |
| UI             | Lucide React Native                          |
| Navigation     | Expo Router 6 (file-based, typed routes)     |
| Push           | OneSignal                                    |
| Error tracking | Sentry                                       |
| Package        | Bun 1.3.9                                    |

## Features

- **Wallet** — Tokens, holdings, pack opening, crypto checkout (via PWA backend)
- **Shop** — Collectibles, cart, coin bundles, checkout
- **Leaderboard** — Rankings and filters
- **Locker room** — Social feed, quests, daily pick'em, spin wheel, and guess-the-player games
- **Profile** — User profile, achievements, followers/following, and settings
- **Notifications** — Push notifications via OneSignal
- **Theme** — Dark/light mode (default dark)

## Backend integration surfaces

- Primary API source: `d-sports-api` under `/api/*`.
- Native app consumes team, league, quest, reward, wallet, commerce, and social endpoints.
- Retry/caching logic and auth token injection are centralized in app API client modules.

## Quest and pass UX integration

- Quest UI now reflects team-scoped eligibility and pass-gated progression where applicable.
- Reward presentation differentiates free/unpaid states and claimability.
- Native progression state is expected to mirror backend eligibility and completion semantics.

## Team-aware experiences

- Team discovery, favorite-team interactions, and leaderboard context are integrated across locker room and profile surfaces.
- Team and league metadata is sourced from backend sync pipelines fed by the `leagues` repository.

## Compliance and legal: pack opening odds

- Native odds disclosure behavior is documented with policy-aligned copy and evidence requirements.
- Disclosure values must come from backend payloads and remain visible/accessible near open actions.

## Getting started

<Steps>
  <Step title="Install dependencies">
    ```bash
    bun install
    ```
  </Step>
  <Step title="Configure environment variables">
    Create a `.env` file in the project root with these `EXPO_PUBLIC_*` keys:

    | Variable | Description |
    | -------- | ----------- |
    | `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` | Clerk publishable key for authentication |
    | `EXPO_PUBLIC_API_URL` | Backend API base URL (e.g. `https://api.d-sports.org`) |
    | `EXPO_PUBLIC_TW_CLIENT_ID` | Thirdweb client ID for Web3 features |
    | `EXPO_PUBLIC_REVENUECAT_API_KEY` | RevenueCat API key for in-app purchases |
    | `EXPO_PUBLIC_REVENUECAT_APPSTORE_ID` | RevenueCat App Store ID |
    | `EXPO_PUBLIC_REVENUECAT_ENTITLEMENT` | RevenueCat entitlement identifier (e.g. `premium`) |
    | `EXPO_PUBLIC_SUPABASE_URL` | Supabase project URL |
    | `EXPO_PUBLIC_SUPABASE_KEY` | Supabase publishable (anon) key |

    <Note>Only `EXPO_PUBLIC_*` prefixed variables are accessible at runtime in the Expo app.</Note>
  </Step>
  <Step title="Start the development server">
    ```bash
    bunx expo start
    ```

    Press `a` for Android, `i` for iOS, or scan the QR code with Expo Go.
  </Step>
</Steps>

## Project structure

```
app/
├── (auth)/              # Login, signup, SSO, password reset
├── (onboarding)/        # New user onboarding flow
├── (tabs)/              # Main tab navigation (wallet, shop, leaderboard, locker room, profile)
├── settings/            # Settings pages with nested modals and tabs
├── notifications.tsx    # Notification center
├── profile/[handle].tsx # Public profile view
└── _layout.tsx          # Root layout with providers and auth protection

components/
├── wallet/              # Wallet sub-components (TokenRow, ActionModal, PackOpeningModal, etc.)
├── shop/                # Shop sub-components (CartModal, CryptoCheckoutModal, etc.)
├── locker-room/         # Feed, quests, games, explore fans/teams
├── profile/             # Achievements, followers, points history modals
├── settings/            # Setting items, sections, and settings modals
├── ui/                  # Reusable primitives (Button, TextField, TutorialOverlay, etc.)

hooks/
├── use-wallet-screen.ts # All wallet state, effects, and handlers
├── use-shop-screen.ts   # All shop state, effects, and handlers
└── use-feed-section.ts  # Social feed logic

lib/api/
├── client.ts            # Base HTTP client with auth token injection
├── wallet-api.ts        # Wallet and token endpoints
├── shop-api.ts          # Shop product endpoints
├── user-api.ts          # User profile and account endpoints
├── checkout-api.ts      # Crypto checkout (calls PWA backend)
├── leaderboard-api.ts   # Leaderboard endpoints
├── locker-room-api.ts   # Social feed endpoints
├── quests-api.ts        # Quest and daily challenge endpoints
├── teams-api.ts         # Team data endpoints
├── collectibles-api.ts  # Collectibles and packs endpoints
├── games-api.ts         # Mini-game endpoints
└── cache.ts             # MMKV cache-first fetching
```

<Tip>
  Screen files under `app/(tabs)/` contain only JSX — all state and business logic lives in dedicated hooks (e.g. `use-wallet-screen.ts`, `use-shop-screen.ts`).
</Tip>

## EAS build profiles

The app uses [EAS Build](https://docs.expo.dev/build/introduction/) with the following profiles defined in `eas.json`:

| Profile | Purpose | Notes |
| ------- | ------- | ----- |
| `development` | Local dev builds | Development client, internal distribution, iOS simulator |
| `development-device` | On-device dev builds | Development client, internal distribution |
| `preview` | Internal testing | Internal distribution, Android APK output |
| `staging` | Pre-production | Auto-increment version |
| `production` | App Store / Play Store release | Auto-increment version |

Common build commands:

```bash
bun run build:dev           # Build dev for all platforms
bun run build:dev:ios       # Build dev for iOS only
bun run build:preview:android  # Build preview APK for Android
bun run build:prod          # Build production for all platforms
```

## CI/CD

A GitHub Actions workflow (`.github/workflows/eas-update.yml`) publishes over-the-air (OTA) updates via EAS Update on pushes to `develop` and `main`:

```bash
eas update --platform ios --branch <branch> --non-interactive
```

This requires an `EXPO_TOKEN` secret configured in the repository.

## Architecture notes

- **Zustand + MMKV** for global state with synchronous persistence
- **React Context** for auth (`UserContext`), collectibles (`CollectiblesContext`), and navbar visibility
- **Crypto checkout** via Thirdweb SDK calling the PWA backend (`POST /api/checkout/crypto`)
- **RevenueCat** handles fiat payments: Apple IAP (native), Google Play (native), Stripe (web)
- **Path alias** `@/*` maps to the project root for cleaner imports
- **PWA-ready** — `display: standalone`, responsive desktop layout (max-width 480 px), web hover states

The app targets both native and web (responsive) and uses the same backend (d-sports-api) as the PWA for API and checkout flows.
