## Overview

**d-sports-engage-native** (package name: `engage-native`) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android: wallet, shop, leaderboard, locker room, and profile.

- **Run:** `bunx expo start` — then press `a` for Android, `i` for iOS, or scan the QR code with Expo Go.
- **Bundle ID:** `com.dsports.wallet.beta`
- **URL scheme:** `engagenative`

## Features

- **Wallet** — Token balances, holdings, pack opening, send/receive crypto, and crypto checkout via the PWA backend
- **Shop** — Collectibles, cart, coin bundles, 3D model previews, and RevenueCat or crypto checkout
- **Leaderboard** — Rankings with filters
- **Locker room** — Social feed, posts, quests, daily games (Pick 'Em, Spin Wheel, Guess the Player), team exploration, and fan discovery
- **Profile** — User profile, followers/following, points history, and settings
- **Settings** — Account, appearance, accessibility, notifications, privacy, data, audio, and support pages
- **Onboarding** — New user onboarding flow with team selection
- **Theme** — Dark/light mode (default dark)

## Environment variables

All runtime variables use the `EXPO_PUBLIC_` prefix. Create a `.env` file in the repo root:

| Variable                              | Purpose                         |
| ------------------------------------- | ------------------------------- |
| `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY`   | Clerk authentication            |
| `EXPO_PUBLIC_API_URL`                 | Backend API base URL            |
| `EXPO_PUBLIC_TW_CLIENT_ID`           | Thirdweb client ID              |
| `EXPO_PUBLIC_REVENUECAT_API_KEY`     | RevenueCat API key              |
| `EXPO_PUBLIC_REVENUECAT_APPSTORE_ID` | RevenueCat App Store ID         |
| `EXPO_PUBLIC_REVENUECAT_ENTITLEMENT` | RevenueCat entitlement name     |
| `EXPO_PUBLIC_SUPABASE_URL`           | Supabase project URL            |
| `EXPO_PUBLIC_SUPABASE_KEY`           | Supabase publishable (anon) key |

## Getting started

1. Clone the repository and run `bun install`.
2. Copy `.env.example` to `.env` and fill in the required values (see table above).
3. Run `bunx expo start` — press `a` for Android, `i` for iOS, or scan the QR code.
4. For a development build: `bun run build:dev` (runs EAS Build for both platforms).

## EAS build profiles

Builds use [EAS Build](https://docs.expo.dev/build/introduction/) with five profiles defined in `eas.json`:

| Profile                | Channel         | Notes                                   |
| ---------------------- | --------------- | --------------------------------------- |
| `development`          | `development`   | Dev client, internal distribution, iOS simulator |
| `development-device`   | `development`   | Dev client on physical device            |
| `preview`              | `preview`       | Internal distribution, Android APK       |
| `staging`              | `staging`       | Auto-increment version                   |
| `production`           | `production`    | Auto-increment version, App Store / Play Store |

Common build commands:

```bash
bun run build:dev           # Development build (all platforms)
bun run build:preview       # Preview build (all platforms)
bun run build:prod          # Production build (all platforms)
bun run build:prod:ios      # Production iOS only
bun run build:prod:android  # Production Android only
```

## Architecture

### Modular screen pattern

Screen files in `app/(tabs)/` contain **only JSX**. All state, effects, and handlers live in dedicated hooks:

- `hooks/use-wallet-screen.ts` — wallet/token fetch, PIN verification, transaction handlers
- `hooks/use-shop-screen.ts` — cart, product queries, carousel auto-scroll, checkout logic

Sub-components are extracted into `components/wallet/` and `components/shop/` with barrel exports.

### API client layer

The API client in `lib/api/` provides domain-specific modules for each feature area:

- `client.ts` — base HTTP client with Clerk auth token injection
- `cache.ts` — MMKV cache-first fetching via `fetchWithCache()`
- Domain modules: `wallet-api`, `shop-api`, `user-api`, `quests-api`, `leaderboard-api`, `locker-room-api`, `teams-api`, `collectibles-api`, `checkout-api`, `games-api`

All API calls go through the shared backend (`d-sports-api`).

### Payments and crypto checkout

- **Fiat:** RevenueCat handles Apple IAP (native), Google Play (native), and Stripe (web) via `lib/revenuecat/provider.tsx`
- **Crypto:** Thirdweb SDK signs on-chain transactions; the app calls the PWA backend at `POST /api/checkout/crypto` and `POST /api/checkout/crypto/verify`. Supported chains: Arbitrum (default), Ethereum, Polygon.

### CI/CD

A GitHub Actions workflow (`.github/workflows/eas-update.yml`) publishes OTA updates via `eas update` on push to `master`.

The app targets both native and web (responsive with a `maxWidth: 480px` container) and uses the same backend (`d-sports-api`) as the PWA.
