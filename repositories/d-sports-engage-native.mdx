## Overview

**d-sports-engage-native** (package name `engage-native`) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android with wallet, shop, leaderboard, locker room, and profile screens. The app also runs on web as a responsive PWA.

## Tech stack

| Category    | Technology                                    |
| ----------- | --------------------------------------------- |
| Framework   | Expo 54, React Native 0.81, React 19          |
| Auth        | Clerk (`@clerk/clerk-expo`)                   |
| Payments    | RevenueCat (Apple IAP, Google Play, Stripe)   |
| Web3        | Thirdweb (on-chain checkout, wallet connect)  |
| State       | Zustand + MMKV persistence                    |
| Navigation  | Expo Router 6 (file-based, typed routes)      |
| Icons       | Lucide React Native                           |
| Monitoring  | Sentry (`@sentry/react-native`)               |
| Backend     | Supabase (client-side), D-Sports API          |
| Package mgr | Bun 1.3                                       |

## Getting started

<Steps>
  <Step title="Clone and install">
    ```bash
    git clone https://github.com/d-sports-ecosystem/d-sports-engage-native.git
    cd d-sports-engage-native
    bun install
    ```
  </Step>
  <Step title="Configure environment">
    Copy the `.env` file and set the required `EXPO_PUBLIC_*` variables (see [environment variables](#environment-variables) below). Only keys prefixed with `EXPO_PUBLIC_` are accessible at runtime.
  </Step>
  <Step title="Start the dev server">
    ```bash
    bunx expo start
    ```
    Press **a** for Android, **i** for iOS, or scan the QR code with Expo Go.
  </Step>
  <Step title="Run a development build (optional)">
    Development builds include `expo-dev-client` for a full dev menu and fast refresh:

    ```bash
    bun run build:dev:ios      # iOS simulator
    bun run build:dev:android  # Android APK
    ```

    After installing the build, start the server with:

    ```bash
    bunx expo start --dev-client
    ```
  </Step>
</Steps>

## Environment variables

All runtime variables use the `EXPO_PUBLIC_` prefix. Set these in your `.env` file:

| Variable                              | Description                     |
| ------------------------------------- | ------------------------------- |
| `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY`   | Clerk publishable key for auth  |
| `EXPO_PUBLIC_API_URL`                 | D-Sports API base URL           |
| `EXPO_PUBLIC_TW_CLIENT_ID`           | Thirdweb client ID for Web3     |
| `EXPO_PUBLIC_REVENUECAT_API_KEY`     | RevenueCat API key              |
| `EXPO_PUBLIC_REVENUECAT_APPSTORE_ID` | RevenueCat App Store ID         |
| `EXPO_PUBLIC_REVENUECAT_ENTITLEMENT` | RevenueCat entitlement name     |
| `EXPO_PUBLIC_SUPABASE_URL`           | Supabase project URL            |
| `EXPO_PUBLIC_SUPABASE_KEY`           | Supabase publishable (anon) key |

<Warning>
  Never commit real secret keys to the repository. The `.env` file in the repo contains development/test values only.
</Warning>

## Architecture

### Routing

The app uses **Expo Router** with file-based routing and typed routes. Route groups split the app into three flows:

- `(auth)` — login, sign-up, SSO callback, password reset
- `(onboarding)` — new user setup
- `(tabs)` — main tab bar (wallet, shop, leaderboard, locker room, profile)

Auth protection in `_layout.tsx` automatically redirects unauthenticated users to login and users who haven't completed onboarding to the onboarding flow.

### Screen pattern

Screen files contain **only JSX**. All state, effects, and handlers live in dedicated hooks:

- `hooks/use-wallet-screen.ts` — wallet balances, token fetching, PIN verification, transaction handling
- `hooks/use-shop-screen.ts` — cart state, product queries, carousel auto-scroll, checkout logic

Sub-components are extracted into `components/wallet/` and `components/shop/` with barrel exports via `index.ts`.

### State management

- **Zustand** (`services/store.ts`) with MMKV persistence for global state (theme, cart, points)
- **React Context** providers for auth/user profile, collectibles, navbar visibility, and accessibility settings
- **MMKV** (`services/storage.ts`) for synchronous key-value storage

### API client

The API client in `lib/api/client.ts` automatically injects Clerk auth tokens into every request. Domain-specific modules (`wallet-api.ts`, `shop-api.ts`, `user-api.ts`, `quests-api.ts`, `leaderboard-api.ts`, etc.) are accessed through the `useApi()` hook exported from `lib/api/index.ts`. A cache-first strategy with MMKV is available via `lib/api/cache.ts`.

### Payments

| Method | Provider    | Platforms                  |
| ------ | ----------- | -------------------------- |
| Fiat   | RevenueCat  | Apple IAP, Google Play, Stripe (web) |
| Crypto | Thirdweb    | Arbitrum (default), Ethereum, Polygon |

Crypto checkout calls the PWA backend (`POST /api/checkout/crypto`) and verifies on-chain via `POST /api/checkout/crypto/verify`.

## EAS builds and deployment

The project uses **EAS (Expo Application Services)** for native builds, OTA updates, and store submissions.

### Build profiles

| Profile         | Channel       | Use case                  |
| --------------- | ------------- | ------------------------- |
| `development`   | `development` | Dev builds with dev client |
| `preview`       | `preview`     | Internal QA distribution   |
| `staging`       | `staging`     | Pre-production testing     |
| `production`    | `production`  | App Store / Play Store     |

### Common commands

```bash
# Development build (iOS simulator)
bun run build:dev:ios

# Preview build (internal distribution)
bun run build:preview

# Production build (store-ready)
bun run build:prod

# OTA update (JS-only changes, no new native build needed)
bun run update --branch production --message "description of changes"

# Submit to stores
bun run submit
```

<Note>
  OTA updates only work when the JS bundle is compatible with the installed native binary. If you add or change native modules, bump the `version` in `app.json` and do a full build first.
</Note>

### CI/CD

A GitHub Actions workflow (`.github/workflows/eas-update.yml`) automatically pushes OTA updates when changes are merged to `main` or `develop`.

## Features

- **Wallet** — token balances, holdings, pack opening, crypto checkout via PWA backend
- **Shop** — collectibles, cart, coin bundles, 3D model viewer, fiat and crypto checkout
- **Leaderboard** — rankings with filters and detail modals
- **Locker room** — social feed, daily pick'em, spin wheel, guess the player, quests, D-Sports Cash
- **Profile** — user profile, team membership, public profile deep links (`/profile/[handle]`)
- **Settings** — account, appearance, notifications, privacy, accessibility, support
- **Theme** — dark/light mode (default dark), brand colors (Gold `#F5C842`, Blue `#4169E1`)
