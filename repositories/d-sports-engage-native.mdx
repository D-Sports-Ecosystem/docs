## Overview

**d-sports-engage-native** (package name: `engage-native`) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android: wallet, shop, leaderboard, locker room, and profile.

- **Run:** `bunx expo start` or `bun run start` — then press `a` for Android or `i` for iOS, or scan the QR code with Expo Go.

## Features

- **Wallet** — Tokens, holdings, pack opening, crypto checkout (via PWA backend), PIN-protected actions
- **Shop** — Collectibles, cart, coin bundles, checkout (fiat via RevenueCat, crypto via Thirdweb)
- **Leaderboard** — Rankings and filters
- **Locker room** — Social feed, quests, daily games (Pick'Em, Spin Wheel, Guess the Player)
- **Profile** — User profile, settings, and team membership
- **Theme** — Dark/light mode (default dark)

## Wallet PIN security

The wallet uses a **6-digit PIN** to protect sensitive actions such as sending transactions and revealing seed phrases. The PIN system includes:

- **PIN on wallet creation** — you set a PIN when creating or importing a wallet.
- **PIN verification** — the app calls `POST /api/wallets/verify-pin` before revealing private keys or signing transactions.
- **Force-reset migration** — if the backend flags a wallet with `pinMigrationRequired`, the app forces a mandatory PIN reset the next time you open the wallet screen. You cannot skip this step.
- **Backfill** — wallets created before PIN support existed are automatically flagged for PIN setup on your next login.

<Note>
  The PIN reset flow is mandatory — the modal cannot be dismissed until you set a new 6-digit PIN. This applies to all wallets flagged with `pinMigrationRequired` by the backend.
</Note>

## Environment variables

The app requires `EXPO_PUBLIC_*` keys in a `.env` file at the project root. Only these prefixed keys are accessible at runtime.

| Variable | Purpose |
| -------- | ------- |
| `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` | Clerk authentication |
| `EXPO_PUBLIC_API_URL` | Backend API base URL (e.g. `https://api.d-sports.org`) |
| `EXPO_PUBLIC_TW_CLIENT_ID` | Thirdweb client ID for web3 features |
| `EXPO_PUBLIC_REVENUECAT_API_KEY` | RevenueCat API key for in-app purchases |
| `EXPO_PUBLIC_REVENUECAT_APPSTORE_ID` | RevenueCat App Store app ID |
| `EXPO_PUBLIC_REVENUECAT_ENTITLEMENT` | RevenueCat entitlement identifier |
| `EXPO_PUBLIC_SUPABASE_URL` | Supabase project URL |
| `EXPO_PUBLIC_SUPABASE_KEY` | Supabase public (anon) key |

## Project structure

```text
app/
├── (auth)/              # Login, signup, password reset
├── (onboarding)/        # New user onboarding flow
├── (tabs)/              # Main tab navigation (wallet, shop, leaderboard, locker room, profile)
├── settings/            # Settings pages with nested modals/tabs
└── _layout.tsx          # Root layout with providers and auth protection

components/
├── wallet/              # Wallet sub-components (token rows, modals, pack opening)
├── shop/                # Shop sub-components (cart, checkout, collectible cards)
├── locker-room/         # Social feed, games, quests
├── leaderboard/         # Leaderboard rows and modals
├── settings/            # Settings items, sections, and modals
├── ui/                  # Reusable primitives (Button, TextField, DatePicker)
└── Icon/                # Icon wrapper using lucide-react-native

hooks/
├── use-wallet-screen.ts # All wallet state, effects, PIN logic, and handlers
├── use-shop-screen.ts   # Cart, checkout, and product logic
└── use-feed-section.ts  # Social feed logic

lib/api/                 # API client modules (wallet, shop, user, quests, etc.)
context/                 # React context providers (user, collectibles, navbar)
services/                # Zustand store, MMKV storage, types
theme/                   # Color system and design tokens
```

## Getting started

<Steps>

    ```bash
    bun install
    ```
  </Step>
  <Step title="Configure environment">

## Architecture notes

- **Modular screen pattern** — screen files contain only JSX; all state and handlers live in dedicated hooks (`use-wallet-screen.ts`, `use-shop-screen.ts`).
- **API client layer** — `lib/api/` modules wrap the backend with auth token injection and MMKV cache fallback.
- **Crypto checkout** — uses Thirdweb SDK to sign on-chain transactions, calling the PWA backend (`POST /api/checkout/crypto`) for payment requests.
- **Payments** — RevenueCat handles fiat payments (Apple IAP, Google Play, Stripe on web).
- **PWA-ready** — responsive desktop layout (`maxWidth: 480px`), web hover states, and `display: standalone` manifest.
- **Path alias** — `@/*` maps to the project root.

The app targets both native and web (responsive) and uses the same backend (d-sports-api) as the PWA for API and checkout flows.
