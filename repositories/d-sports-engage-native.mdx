## Overview

**d-sports-engage-native** (package name: `engage-native`) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android: wallet, shop, leaderboard, locker room, and profile. The app also runs as a responsive PWA on desktop web.

## Tech stack

| Category       | Technology                                     |
| -------------- | ---------------------------------------------- |
| Framework      | Expo 54, React Native 0.81, React 19           |
| Auth           | Clerk (`@clerk/clerk-expo`)                    |
| Payments       | RevenueCat (`react-native-purchases`)          |
| Web3           | Thirdweb SDK                                   |
| State          | Zustand 5 with MMKV persistence                |
| Storage        | MMKV (`react-native-mmkv`)                     |
| UI             | Lucide React Native                            |
| Navigation     | Expo Router 6 (file-based, typed routes)       |
| 3D             | React Three Fiber, Filament (native GLB viewer)|
| Monitoring     | Sentry (`@sentry/react-native`)                |
| Package manager| Bun 1.3.9                                      |

## Features

- **Wallet** — tokens, holdings, pack opening, crypto checkout (via d-sports-api backend)
- **Shop** — collectibles, cart, coin bundles, fiat and crypto checkout
- **Leaderboard** — rankings with filters
- **Locker room** — social feed, quests, daily games (Pick'Em, Spin Wheel, Guess the Player)
- **Profile** — user profile, settings, and team membership
- **Theme** — dark/light mode (default dark)

- Team-aware experiences were expanded to align with backend team/favorites behavior.
- Quest and rewards surfaces were updated to support per-team progression and pass-aware eligibility.

## Backend integration surfaces

- Primary API source: `d-sports-api` under `/api/*`.
- Native app consumes team, league, quest, reward, wallet, commerce, and social endpoints.
- Retry/caching logic and auth token injection are centralized in app API client modules.

## Quest and pass UX integration

- Quest UI now reflects team-scoped eligibility and pass-gated progression where applicable.
- Reward presentation differentiates free/unpaid states and claimability.
- Native progression state is expected to mirror backend eligibility and completion semantics.

## Team-aware experiences

- Team discovery, favorite-team interactions, and leaderboard context are integrated across locker room and profile surfaces.
- Team and league metadata is sourced from backend sync pipelines fed by the `leagues` repository.

## Compliance and legal: pack opening odds

- Native odds disclosure behavior is documented with policy-aligned copy and evidence requirements.
- Disclosure values must come from backend payloads and remain visible/accessible near open actions.

## Getting started

<Steps>
  <Step title="Install dependencies">
    ```bash
    bun install
    ```
  </Step>
  <Step title="Configure environment variables">
    Create a `.env` file at the project root. Only `EXPO_PUBLIC_*` keys are accessible at runtime.

    | Variable | Purpose |
    | -------- | ------- |
    | `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` | Clerk authentication |
    | `EXPO_PUBLIC_API_URL` | d-sports-api backend URL |
    | `EXPO_PUBLIC_TW_CLIENT_ID` | Thirdweb client ID (Web3) |
    | `EXPO_PUBLIC_REVENUECAT_API_KEY` | RevenueCat API key |
    | `EXPO_PUBLIC_REVENUECAT_APPSTORE_ID` | RevenueCat App Store ID |
    | `EXPO_PUBLIC_REVENUECAT_ENTITLEMENT` | RevenueCat entitlement identifier |
    | `EXPO_PUBLIC_SUPABASE_URL` | Supabase project URL |
    | `EXPO_PUBLIC_SUPABASE_KEY` | Supabase publishable key |
  </Step>
  <Step title="Start the development server">
    ```bash
    bunx expo start
    ```
    Press **a** for Android, **i** for iOS, or scan the QR code with Expo Go.
  </Step>
</Steps>

## EAS build profiles

The project uses [EAS Build](https://docs.expo.dev/build/introduction/) with the following profiles defined in `eas.json`:

| Profile | Channel | Notes |
| ------- | ------- | ----- |
| `development` | `development` | Dev client, internal distribution, iOS simulator enabled |
| `development-device` | `development` | Dev client on physical devices |
| `preview` | `preview` | Internal distribution, Android APK |
| `staging` | `staging` | Auto-increment version |
| `production` | `production` | Auto-increment version, App Store / Play Store |

## CI/CD

A GitHub Actions workflow (`.github/workflows/eas-update.yml`) runs on every push to `master`. It publishes an OTA update via `eas update` to the matching branch channel (`develop` or `main`).

```
app/
├── (auth)/              # Login, signup, SSO, password reset
├── (onboarding)/        # New-user onboarding flow
├── (tabs)/              # Main tab navigation (wallet, shop, leaderboard, locker room, profile)
├── settings/            # Settings pages with nested modals and tabs
└── _layout.tsx          # Root layout with providers and auth protection

components/
├── wallet/              # Wallet sub-components (TokenRow, ActionModal, PackOpeningModal, …)
├── shop/                # Shop sub-components (CartModal, CryptoCheckoutModal, …)
├── locker-room/         # Locker room components (quests, games, feed, teams)
├── leaderboard/         # Leaderboard components
├── settings/            # Settings items, modals, and tabs
├── ui/                  # Reusable primitives (Button, TextField, TutorialOverlay, …)
├── layout/              # AppScreen responsive wrapper
└── Icon/                # Icon wrapper using lucide-react-native

hooks/                   # Screen-level hooks (use-wallet-screen, use-shop-screen, …)
lib/api/                 # API client modules per domain (wallet, shop, user, quests, …)
lib/revenuecat/          # RevenueCat in-app purchases provider
context/                 # React Context providers (user, collectibles, navbar, actions)
services/                # Zustand store, MMKV storage adapter, shared types
theme/                   # Color system, spacing, typography tokens
types/                   # Shared TypeScript types (wallet, shop, checkout, API)
constants/               # Shared constants (tokens, coin bundles, wallet providers)
```

## Architecture patterns

- **Modular screen architecture** — screen files contain only JSX. All state, effects, and handlers live in dedicated hooks (`use-wallet-screen.ts`, `use-shop-screen.ts`).
- **Extracted sub-components** — wallet and shop screens are decomposed into `components/wallet/` and `components/shop/` with barrel exports via `index.ts`.
- **API client layer** — `lib/api/client.ts` provides a base HTTP client with auth token injection. Domain modules (`wallet-api.ts`, `shop-api.ts`, etc.) are accessed through the `useApi()` hook. Responses are cached with MMKV via `lib/api/cache.ts`.
- **Dual checkout** — fiat payments use RevenueCat (Apple IAP, Google Play, Stripe on web). Crypto checkout calls the d-sports-api backend and signs on-chain transactions with Thirdweb on Arbitrum, Ethereum, or Polygon.
- **Path alias** — import with `@/*` which maps to the project root.
- **PWA-ready** — `display: standalone`, responsive `maxWidth: 480px` container on desktop, web hover states, and keyboard navigation.
