## Overview

**d-sports-engage-native** (package name: `engage-native`, current version **1.13.0**) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android: wallet, shop, leaderboard, locker room, and profile.

- **Run:** `bunx expo start` — then press `a` for Android, `i` for iOS, or scan the QR code with Expo Go.

## Tech stack

| Category   | Technology                                    |
| ---------- | --------------------------------------------- |
| Framework  | Expo 54, React Native 0.81, React 19          |
| Auth       | Clerk (Expo)                                  |
| Payments   | RevenueCat (`react-native-purchases`)         |
| Web3       | Thirdweb                                      |
| State      | Zustand + MMKV persistence                    |
| Push       | OneSignal (iOS, Android, web PWA)             |

| UI         | Lucide React Native                           |
| Navigation | Expo Router (file-based, typed routes)        |
| Package    | Bun                                           |

## Features

- **Wallet** — BIP39 wallet creation/import, token holdings, multichain balance reads (Ethereum, Polygon, BSC), send/receive with QR, pack opening, crypto checkout via PWA backend
- **Shop** — Collectibles, cart, coin bundles, 3D/GLB model viewing, crypto and fiat checkout (RevenueCat for Apple IAP / Google Play / Stripe)
- **Leaderboard** — Rankings and filters
- **Locker room** — Social feed, posts, reactions, quests, daily pick'em, daily spin wheel, and explore fans/teams
- **Profile** — User profile with follow system, avatar/banner upload, stats modals (points history, followers, following)
- **Settings** — API-backed settings for appearance, notifications, privacy, accessibility, room, and data preferences; blocked users management; in-app support tickets; about/changelog viewer
- **Theme** — Dark/light mode (default dark)

## Environment variables

The app uses `EXPO_PUBLIC_*` variables, which are accessible at runtime. Create a `.env` file at the repo root:

```bash
# Clerk authentication
EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY="pk_live_..."

# API backend
EXPO_PUBLIC_API_URL="https://api.d-sports.org"

# Thirdweb (Web3)
EXPO_PUBLIC_TW_CLIENT_ID="your-thirdweb-client-id"

# RevenueCat (in-app purchases)
EXPO_PUBLIC_REVENUECAT_API_KEY="rcb_..."
EXPO_PUBLIC_REVENUECAT_APPSTORE_ID="appl_..."
EXPO_PUBLIC_REVENUECAT_ENTITLEMENT="premium"

# Supabase (client-side features — NOT direct DB)
EXPO_PUBLIC_SUPABASE_URL="https://your-project.supabase.co"
EXPO_PUBLIC_SUPABASE_KEY="sb_publishable_..."
```

## Architecture

### Routing

The app uses **Expo Router** with file-based routing and typed routes (`experiments.typedRoutes: true`):

- `app/(tabs)/` — Main tab navigation (wallet, shop, leaderboard, locker room, profile)
- `app/(auth)/` — Login, sign-up, password reset, SSO callback
- `app/(onboarding)/` — New user onboarding flow
- `app/settings/` — Settings pages with nested modals and tabs
- `app/profile/[handle].tsx` — Public profile by handle

Auth protection in `_layout.tsx` automatically redirects based on authentication and onboarding state.

### Screen pattern

Screen files in `app/(tabs)/` are **thin render shells** — they contain only imports and JSX. All state, effects, and handlers live in dedicated hooks:

- `hooks/use-wallet-screen.ts` — Wallet business logic (balance fetches, PIN verification, transactions)
- `hooks/use-shop-screen.ts` — Shop business logic (cart, products, checkout)

Sub-components are extracted into `components/<domain>/` directories with barrel `index.ts` exports.

### State management

- **Zustand** store in `services/store.ts` with MMKV persistence
- **React Context** providers: `UserContext` (auth/profile), `CollectiblesContext` (packs/items), `NavBarVisibilityContext`, `CreateActionContext`

### API client

- `lib/api/client.ts` — Base HTTP client with auth token injection
- `lib/api/index.ts` — `useApi()` hook exposing all API modules
- Domain modules: `wallet-api`, `shop-api`, `user-api`, `quests-api`, `leaderboard-api`, `locker-room-api`, `teams-api`, `collectibles-api`, `checkout-api`
- `lib/api/cache.ts` — `fetchWithCache()` MMKV cache-first fetching

### Payments

| Method | Provider | Platforms |
| ------ | -------- | --------- |
| Apple IAP | RevenueCat | iOS |
| Google Play | RevenueCat | Android |
| Stripe | RevenueCat | Web |
| Crypto | Thirdweb + PWA backend | Native (Arbitrum, Ethereum, Polygon) |

## EAS build and OTA updates

The project uses **EAS Build** and **EAS Update** for native builds and over-the-air updates.

### Build profiles

| Profile       | Use case              | Distribution |
| ------------- | --------------------- | ------------ |
| `development` | Dev client (simulator) | Internal    |
| `preview`     | Internal testing (APK on Android) | Internal |
| `staging`     | Pre-production        | Internal     |
| `production`  | App Store / Play Store | Store       |

### Common commands

```bash
# Development build
bun run build:dev:ios
bun run build:dev:android

# Preview build
bun run build:preview:ios
bun run build:preview:android

# Production build
bun run build:prod:ios
bun run build:prod:android

# OTA update
bun run update

# Submit to stores
bun run submit:ios
bun run submit:android
```

### CI/CD

A GitHub Actions workflow (`.github/workflows/eas-update.yml`) automatically publishes OTA updates to the `develop` or `main` branch channel on push.

The app targets both native and web (responsive) and uses the same backend (`d-sports-api`) as the PWA for API and checkout flows.
