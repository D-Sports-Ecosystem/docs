## Overview

**d-sports-engage-native** (package name: `engage-native`, version `1.10.10`) is the native mobile app for D-Sports. It mirrors the core PWA experience on iOS and Android: wallet, shop, leaderboard, locker room, and profile. The app also runs as a PWA on the web with a responsive desktop layout.

## Tech stack

| Category    | Technology                                    |
| ----------- | --------------------------------------------- |
| Framework   | Expo 54, React Native 0.81, React 19          |
| Auth        | Clerk (`@clerk/clerk-expo`)                    |
| Payments    | RevenueCat (`react-native-purchases`)          |
| Web3        | Thirdweb SDK                                   |
| State       | Zustand 5 + React Context                      |
| Storage     | MMKV (synchronous persistence)                 |
| Backend     | Supabase (storage/assets), d-sports-api        |
| Monitoring  | Sentry (`@sentry/react-native`)                |
| UI          | Lucide React Native                            |
| Navigation  | Expo Router 6 (file-based, typed routes)       |
| Package     | Bun                                            |

## Features

- **Wallet** — Token balances, holdings, pack opening, crypto checkout (via PWA backend), token logos loaded from Supabase `crypto-token-assets` bucket
- **Shop** — Collectibles, cart, coin bundles, crypto and fiat checkout
- **Leaderboard** — Rankings and filters
- **Locker room** — Social feed, daily games (Pick 'Em, Spin Wheel, Guess the Player), quests
- **Profile** — User profile, settings, team membership
- **Theme** — Dark/light mode (default dark)
- **PWA** — `display: standalone` with responsive max-width container on desktop web

## Getting started

<Steps>
  <Step title="Install dependencies">
    ```bash
    bun install
    ```
  </Step>
  <Step title="Configure environment">
    Create a `.env` file in the project root with the following `EXPO_PUBLIC_*` variables:

    | Variable | Description |
    | -------- | ----------- |
    | `EXPO_PUBLIC_CLERK_PUBLISHABLE_KEY` | Clerk publishable key for authentication |
    | `EXPO_PUBLIC_API_URL` | d-sports-api base URL (e.g. `https://api.d-sports.org`) |
    | `EXPO_PUBLIC_TW_CLIENT_ID` | Thirdweb client ID for Web3 features |
    | `EXPO_PUBLIC_REVENUECAT_API_KEY` | RevenueCat API key |
    | `EXPO_PUBLIC_REVENUECAT_APPSTORE_ID` | RevenueCat App Store ID |
    | `EXPO_PUBLIC_REVENUECAT_ENTITLEMENT` | RevenueCat entitlement identifier |
    | `EXPO_PUBLIC_SUPABASE_URL` | Supabase project URL (used for token asset storage) |
    | `EXPO_PUBLIC_SUPABASE_KEY` | Supabase publishable anon key |

    <Note>Only `EXPO_PUBLIC_*` keys are accessible at runtime in the Expo app.</Note>
  </Step>
  <Step title="Start the dev server">
    ```bash
    bunx expo start
    ```
    Press `a` for Android, `i` for iOS, or scan the QR code with Expo Go.
  </Step>
  <Step title="Type-check">
    ```bash
    bun tsc --noEmit
    ```
  </Step>
</Steps>

## EAS build profiles

Builds use [EAS Build](https://docs.expo.dev/build/introduction/) via `bunx eas-cli`. Profiles are defined in `eas.json`:

| Profile | Command | Distribution | Notes |
| ------- | ------- | ------------ | ----- |
| `development` | `bun run build:dev` | Internal | Dev client, iOS simulator enabled |
| `development-device` | `bunx eas-cli build --profile development-device` | Internal | Dev client on physical device |
| `preview` | `bun run build:preview` | Internal | Android APK build |
| `staging` | `bunx eas-cli build --profile staging` | — | Auto-increment version |
| `production` | `bun run build:prod` | — | Auto-increment version |

OTA updates use `bunx eas-cli update` with channel-based routing.

## Architecture patterns

### Modular screens

Screen files under `app/(tabs)/` contain **only JSX**. All state, effects, and handlers live in dedicated hooks (e.g. `hooks/use-wallet-screen.ts`, `hooks/use-shop-screen.ts`). Sub-components are extracted into `components/wallet/` and `components/shop/` with barrel exports.

### API client layer

The `lib/api/` directory provides domain-specific API modules (`wallet-api.ts`, `shop-api.ts`, `user-api.ts`, etc.) consumed through a `useApi()` hook. The base HTTP client in `lib/api/client.ts` injects the Clerk auth token automatically. Responses are cached via MMKV in `lib/api/cache.ts` for offline fallback.

### Crypto token assets

Token logos are loaded from the Supabase `crypto-token-assets` storage bucket. The helper `getCryptoTokenLogoUrl()` in `lib/crypto-token-assets.ts` maps token symbols (ETH, SOL, WBTC, etc.) to public Supabase storage URLs with an initials-based fallback in the UI.

### Crypto checkout

On-chain payments use the Thirdweb SDK to prepare and sign transactions, calling the PWA backend at `POST /api/checkout/crypto` and `POST /api/checkout/crypto/verify`. Supported chains: Arbitrum (default), Ethereum, and Polygon.

### State management

- **Zustand** (`services/store.ts`) with MMKV persistence for global state (theme, cart, points)
- **React Context** for auth (`UserContext`), collectibles, navbar visibility, and action modals
- Path alias `@/*` maps to the project root
